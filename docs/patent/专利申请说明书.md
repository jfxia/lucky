# 说明书

## 技术领域

本发明属于人工智能与计算机软件技术领域，具体涉及一种面向AI智能体（AI Agent）的声明式编排语言及其配套的编译器、运行时引擎和后端适配系统，用于定义、管理和执行多个AI智能体协同完成复杂目标的工作流。

## 背景技术

随着大语言模型（Large Language Model, LLM）技术的快速发展，AI智能体（AI Agent）已成为人工智能应用的重要形态。AI智能体是指具备自主感知、推理、规划和行动能力的AI实体，通常包含模型配置、工具调用、记忆管理和权限控制等要素。在实际应用中，单个智能体的能力有限，往往需要多个专业化的智能体协同工作才能完成复杂的业务目标。因此，如何高效地编排和管理多个AI智能体的协同执行，成为亟待解决的技术问题。

### 现有技术方案及其不足

**现有技术方案一：基于代码的SDK调用方式。** 开发人员通过通用编程语言（如Python、TypeScript）调用LLM供应商提供的软件开发工具包（SDK），在程序代码中手动管理多个智能体的调用顺序、数据传递、状态同步和错误处理。该方式的代表性框架包括LangChain、LlamaIndex、CrewAI等。其不足在于：（1）需要大量样板代码来实现智能体间的上下文传递和状态管理，开发效率低；（2）LLM调用在语言层面被视为普通函数调用，编译器无法对AI特性进行类型检查、成本估算或优化；（3）多个智能体的并发控制和依赖关系管理完全依赖于开发者的编程水平，容易引入并发错误；（4）缺乏统一的权限管理和审计机制。

**现有技术方案二：基于可视化配置的工作流编排。** 通过图形化界面或配置文件（如YAML、JSON）定义智能体的执行顺序，由工作流引擎解析配置并按序执行。该方式的代表性平台包括Dify、Coze、扣子等。其不足在于：（1）配置格式的表达能力有限，难以描述复杂的控制流（如条件分支、循环、嵌套并行、声明式错误恢复）；（2）缺乏对智能体细粒度属性的统一描述能力，如权限、记忆、推理策略等通常需要在外部分别配置；（3）缺乏编译时验证，配置错误只能在运行时被发现；（4）工作流定义与具体平台绑定，缺乏跨平台的可移植性。

**现有技术方案三：基于自然语言的全自主Agent。** 通过自然语言指令让一个主控Agent自行分解任务、规划步骤并调用其他工具或子Agent。该方式的代表性项目包括AutoGPT、MetaGPT、BabyAGI等。其不足在于：（1）执行过程高度不确定，难以保证可重复性和可审计性；（2）缺乏显式的权限控制边界，存在安全风险；（3）难以对执行效率进行精确的优化和成本控制。

综上所述，现有技术在AI智能体编排方面存在以下共性技术问题：（1）编排逻辑与AI特性之间缺乏语言级的抽象映射；（2）缺乏声明式的、可编译验证的编排语法；（3）智能体间的上下文传递需要大量手动编码；（4）编排程序与具体AI平台绑定，缺乏可移植性。

## 发明内容

### 发明目的

本发明旨在克服现有技术的上述不足，提供一种基于目标导向编排语言（Goal-Oriented Agent Language）的AI智能体协同执行方法及系统，使开发者能够以声明式、接近自然语言的方式定义多智能体的协同工作流，同时保持执行的确定性、安全性、可移植性和可优化性。

### 技术方案

本发明提供一种基于目标导向编排语言的AI智能体协同执行方法，包括以下步骤：

**步骤S1：接收以声明式编排语言编写的源程序。** 所述源程序采用基于缩进的语法结构，包含以下六层语义对象的声明：（a）目标（Goal），用于声明工作完成的成功标准；（b）工作流（Workflow），用于以有向无环图（DAG）的形式定义执行步骤及其依赖关系；（c）智能体（Agent），用于定义具有模型、工具、记忆、权限和提示词的AI执行实体；（d）任务（Task），定义最小的可调度、可检查点的工作单元；（e）操作（Operation），语言内置的AI编排原语；（f）产物（Artifact），每次执行产生的不可变输出。

**步骤S2：将所述源程序编译为中间表示（IR）。** 编译过程包括词法分析、语法分析、语义分析和IR生成。所述IR采用有向无环图（DAG）结构，其中节点代表执行单元，包括但不限于：目标节点（GoalNode）、工作流节点（WorkflowNode）、任务节点（TaskNode）、智能体调用节点（AgentInvokeNode）、工具调用节点（ToolNode）、LLM调用节点（LLMCallNode）、决策节点（DecisionNode）、并行节点（ParallelNode）、汇合节点（JoinNode）、审批节点（ApprovalNode）和错误处理节点（AttemptNode）；边代表节点间的依赖关系，包括控制依赖、数据依赖和资源依赖。

**步骤S3：通过运行时引擎执行所述IR。** 所述运行时引擎包括：（a）DAG调度器，采用基于优先级的拓扑遍历算法，将无依赖节点放入就绪队列，按优先级调度到执行槽；（b）上下文管理器，采用不可变分层结构，使上下文从工作流作用域自动向下传播至智能体和任务作用域；（c）智能体记忆管理器，提供基于键值存储和向量相似性搜索的持久化记忆；（d）权限执行器，基于声明式权限规则进行能力安全检查；（e）工具注册与执行器，通过适配器模式支持多种外部工具的调用。

其中，所述DAG调度器的执行逻辑包括：初始化时，将所有入度为零的节点标记为就绪状态并加入优先级队列；在每轮调度中，从就绪队列中选择优先级最高的节点分配执行槽；节点执行完成后，遍历其后继节点，将后继节点的未满足依赖计数减一；若后继节点的所有依赖均满足，则将其标记为就绪并加入优先级队列；重复上述过程直至所有节点完成或发生终止性失败。其中，节点的优先级根据其在DAG中的关键路径深度和成本估算动态计算。

**步骤S4：通过后端适配器将IR指令映射到具体的AI平台执行。** 后端适配器通过标准化的Lucky工具协议（LTP）与AI平台通信，所述LTP基于JSON-RPC 2.0协议，支持会话管理、IR加载、执行控制、事件流、审批交互和检查点操作。

### 核心技术创新

**创新点一：六层语义抽象模型。** 本发明首次提出了"目标→工作流→智能体→任务→操作→产物"的六层语义抽象模型，将AI智能体编排的抽象层次从传统的代码级别提升到语义级别。其中，Goal层定义"做什么"，Workflow层定义"怎么做"，Agent层定义"谁来做"，Task层定义"做哪些步骤"，Operation层提供内置原语，Artifact层规约输出产物。该分层模型使得开发者可以关注业务语义而非实现细节。

**创新点二：声明式工作流语法。** 本发明设计了基于缩进的声明式工作流语法，其中：（a）使用`->`（连字符加大于号）箭头表示顺序依赖关系；（b）相同缩进级别且无箭头连接的节点默认为并行执行；（c）提供`parallel`/`wait`关键字实现显式的并行-汇合控制；（d）提供`if/else`关键字实现条件分支；（e）提供`attempt/recover/retry/fallback/human`关键字实现声明式错误恢复链。该语法使得工作流的结构与执行逻辑一目了然。

**创新点三：AI模型作为语言原语。** 本发明将大语言模型视为编程语言的基本组成部分而非外部库。语法中通过`model`关键字声明模型配置（包括提供商、版本、温度、最大Token数），通过`use`关键字进行模型选择，通过`ask`关键字进行LLM内联调用，通过`prompt`关键字定义结构化提示词模板。编译器能够理解这些声明，进行类型检查、置信度评估和优化（如模型路由、响应缓存）。

**创新点四：上下文自动传播机制。** 本发明实现了上下文在六层语义抽象之间的自动传播：工作流声明的上下文自动被其内部的所有智能体继承；智能体的上下文自动被其内的所有任务继承；任务可以声明额外的上下文条目，这些条目仅在该任务的作用域内可见。上下文采用不可变分层实现：每个作用域创建一个新的上下文层，子层可以覆盖（shadow）父层的条目，但不可修改父层。这种设计确保了执行过程的可检查点化和可重放性。

**创新点五：多层次的中间表示（IR）架构。** 本发明设计了三个层次的IR：（a）高层IR（HIR）对应Agent/Task粒度的执行DAG，包含18种节点类型和8种边类型；（b）中层IR（MIR）为SSA（静态单赋值）形式的控制流图，支持30余种指令操作码和标准编译器优化遍（死代码消除、常量折叠、公共子表达式消除、拷贝传播、循环不变量外提）；（c）低层IR（LIR）为线性指令序列，可直接解释执行。该多层次架构使得同一编排程序可以在不同粒度和优化级别上进行处理，并可通过标准化协议映射到不同的AI平台后端执行。

**创新点六：基于协议的跨平台执行架构。** 本发明设计了Lucky工具协议（LTP），使得编译后的IR可以在多种AI编码平台上一致执行。LTP基于JSON-RPC 2.0协议，定义了会话管理、IR加载与验证、同步/异步执行、事件流推送、人工审批交互、检查点创建与恢复等标准接口。当前支持的AI平台包括Claude Code、Codex CLI、OpenCode、Cursor和Dify，各平台通过实现LTP客户端适配器来集成Lucky运行时。

### 有益效果

与现有技术相比，本发明具有以下有益效果：

1. **开发效率显著提升。** 通过声明式语法、上下文自动传播和语言级AI抽象，开发者用数十行Lucky代码即可实现需要数百行Python/TypeScript代码才能完成的多智能体协同逻辑。

2. **执行确定性增强。** 尽管AI推理具有概率性，但编排结构（DAG拓扑、调度策略、上下文传播）是完全确定性的。同一编排程序在相同输入下产生相同的执行图，确保了可重复性和可审计性。

3. **后端可移植性。** 通过IR抽象层和LTP协议，同一编排程序无需修改即可在不同的AI平台上执行，实现了"一次编写，到处运行"（Write Once, Run Anywhere）的AI编排体验。

4. **安全性增强。** 通过声明式权限控制（allow/deny）和词法作用域继承，实现最小权限原则。人工审批门（approval）作为语言特性内建于编排流程中，确保关键操作的人类监管。

5. **成本可优化。** 运行时引擎支持模型路由（简单问题使用廉价模型）、LLM响应缓存（相同提示复用缓存结果）和批量处理，可显著降低LLM调用的Token消耗和费用。

6. **可扩展性。** 工具适配器采用Trait模式，支持动态注册新的工具和模型后端。LTP协议支持自定义工具提供者，可集成任意外部系统。

### 实施例1：基于本发明的多智能体代码审查系统

本实施例展示如何使用本发明编排语言实现一个自动化的代码审查（Code Review）系统。该系统在收到Pull Request时，自动调用安全审查智能体和代码风格审查智能体并行执行审查，然后由报告聚合智能体汇总结果并发布评论。

**步骤1：定义模型和智能体。**

```lucky
project CodeReviewBot

use DeepSeek

model DeepSeek(
    provider = "deepseek",
    version = "deepseek-v4",
    temperature = 0.2,
)

agent SecurityReviewer
    model DeepSeek
    tools Git, Filesystem
    permissions
        allow git.diff
        allow git.comment
    prompt SecurityPrompt

agent StyleReviewer
    model DeepSeek
    tools Git, Filesystem
    permissions
        allow git.diff
        allow git.comment

agent ReportAggregator
    model DeepSeek
    tools Git
    permissions
        allow git.comment
```

**步骤2：定义工作流。**

```lucky
workflow FullReview
    context
        pr_number: Int
        base_branch: String = "main"

    FetchPR(pr_number = context.pr_number)
        ->
    parallel
        SecurityReviewer.review(diff = context.diff)
        StyleReviewer.review(diff = context.diff)
    wait
        ->
    ReportAggregator.merge(
        security = context.security_findings,
        style = context.style_findings,
    )
        ->
    PostComment(pr_number = context.pr_number)
```

在该工作流中：(a) FetchPR首先执行，获取PR的差异内容，其输出自动成为上下文的一部分；(b) SecurityReviewer和StyleReviewer并行执行，各自从上下文获取差异内容进行审查；(c) `wait`关键字构成同步屏障，确保两个审查都完成后才继续；(d) ReportAggregator汇总两个审查的结果；(e) PostComment将汇总结果发布为PR评论。

**步骤3：编译与执行。**

开发者使用命令行工具`lucky run main.lk`即可编译并执行上述编排程序。编译器自动完成词法分析、语法分析、语义分析和IR生成。运行时引擎加载IR后，按DAG拓扑顺序调度执行：先执行FetchPR节点（优先级较低），完成后同时调度SecurityReviewer和StyleReviewer节点（并行），两个节点都完成后调度ReportAggregator节点，最后执行PostComment节点。

### 实施例2：基于本发明的带自动回滚的数据库迁移系统

本实施例展示如何使用本发明的声明式错误恢复机制实现安全的数据库迁移。

```lucky
project DatabaseMigration

use DeepSeek

model DeepSeek(
    provider = "deepseek",
    version = "deepseek-v4",
    temperature = 0.0,
)

agent MigrationRunner
    model DeepSeek
    tools Database, Filesystem
    permissions
        allow database.execute
        allow filesystem.read
    policy
        timeout 30m
        checkpoint before retry

workflow RunMigrations
    context
        migrations_dir: String = "./migrations"

    DiscoverMigrations(migrations_dir = context.migrations_dir)
        ->
    CreateBackup(tables = ["*"])
        ->
    for migration in context.pending
        ValidateMigration(migration = migration)
            ->
        attempt
            RunMigration(migration = migration)
        recover
            retry 2 with backoff exponential(max: 5m)
        recover
            Shell.exec("psql -f " + context.backup_path)
            human escalate "Migration failed and was rolled back"
```

本实施例展示了：(a)`for`循环遍历待执行的迁移；(b)`attempt/recover`声明式错误恢复：迁移失败时先重试2次（带指数退避），若仍失败则自动执行数据库恢复脚本，并升级给人工处理。

### 实施例3：编译器的具体实现

本发明的编译器采用Rust编程语言实现，主要包括以下模块：

1. **词法分析器（Lexer）**：手写状态机实现，支持Python风格的INDENT/DEDENT缩进处理。Token类型包括关键字（50+）、标识符、字面量（整数、浮点、字符串、布尔、null、unknown）、运算符（含`->`、`|>`等复合运算符）、注释和缩进标记。

2. **语法分析器（Parser）**：采用递归下降（Recursive Descent）方法，表达式解析使用Pratt算法处理运算符优先级。支持完整的声明式编排语法，包括Agent声明（含model/memory/tools/permissions/policy/prompt六个属性段）、Task声明（含input/output/steps/context/policy/rollback六个属性段）、Workflow声明（含`->`箭头和`parallel`/`wait`并行控制）、Goal声明（含success成功标准和workflow引用）。

3. **语义分析器（Semantic Analyzer）**：实现基于作用域栈的符号表和名称解析，支持12种符号类型（Agent、Task、Workflow、Goal、Memory、Tool、Model、Prompt、Policy、Type、Variable、Parameter）。

4. **HIR生成器（HIR Builder）**：将带类型标注的AST转换为执行DAG，生成18种节点类型和8种边类型。

5. **优化器（Optimizer）**：支持5种优化遍——死代码消除（DCE）、常量折叠、本地公共子表达式消除（CSE）、拷贝传播和循环不变量外提（LICM）。

### 实施例4：运行时引擎的核心数据结构

运行时引擎的核心数据结构包括：

1. **NodeState（节点状态）**：记录每个执行节点的当前状态（Pending/Ready/Running/Completed/Failed/Cancelled/Skipped）、输出值、所属智能体名称、重试计数、深度（在DAG中距离入口节点的距离）和未满足的入边计数。

2. **PriorityQueue（优先级队列）**：基于节点深度和关键路径权重的优先级队列。节点优先级的计算公式为：$\text{Priority} = f(\text{depth}, \text{critical\_path}, \text{estimated\_cost})$，其中关键路径上的节点获得更高优先级。

3. **ContextLayer（上下文层）**：不可变键值对映射，包含指向父层的引用。读取操作沿父链向上查找，写入操作在本地层创建新条目（不修改父层）。

4. **AgentMemory（智能体记忆）**：包含键值存储（BTreeMap）和向量索引（VectorIndex）。向量索引采用暴力搜索（Brute-force cosine similarity），支持K近邻（K-NN）语义搜索。

5. **PermissionSet（权限集）**：包含允许列表和拒绝列表，拒绝规则优先于允许规则，默认策略为拒绝。
